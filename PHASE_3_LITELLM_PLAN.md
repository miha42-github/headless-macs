# Phase 3 — LiteLLM Integration Plan

## Executive Summary

LiteLLM is a proxy/gateway layer, **not an inference engine**. It sits in front of Ollama, Rapid-MLX,
mlx-lm, and cloud providers (Anthropic, OpenAI, Groq, Bedrock) and exposes a single OpenAI-compatible
endpoint. It does not replace any inference tool — it routes to them.

**The key value proposition for this project:** coding agents (Claude Code, Cursor, Aider) always point
to `http://localhost:4000`. When you swap a backend (Ollama → Rapid-MLX, local → cloud), you edit one
YAML file and restart one daemon. The client config never changes.

**Current version:** v1.88.0 (June 6, 2026)  
**Default port:** 4000  
**Install:** `uv tool install 'litellm[proxy]'`

---

## Architecture: Where LiteLLM Fits

```
Coding agent / Claude Code / Cursor
         │
         ▼  http://localhost:4000  (LiteLLM proxy)
┌────────────────────────────────────────────────────┐
│             litellm_config.yaml                    │
│                                                    │
│  "code"    → Rapid-MLX  :8000  (primary)          │
│            → Ollama      :11434 (fallback)         │
│            → Groq cloud          (fallback)        │
│                                                    │
│  "embed"   → Infinity   :7997                      │
│                                                    │
│  "reason"  → Ollama deepseek-r1 :11434            │
│            → Anthropic claude-3-7-sonnet (fallback)│
└────────────────────────────────────────────────────┘
         │              │              │
         ▼              ▼              ▼
    Rapid-MLX       Ollama         Infinity
     :8000          :11434          :7997
```

LiteLLM adds **no inference** — it routes and translates format. The existing inference tools
(Ollama, Rapid-MLX, mlx-lm, Infinity, Exo) continue to run exactly as before.

## Pro / Con: LiteLLM Layer vs Direct Ollama

### When LiteLLM Adds Clear Value

| Scenario | Without LiteLLM | With LiteLLM |
|---|---|---|
| Swap backend (Ollama → Rapid-MLX) | Update URL in every client tool | Edit litellm_config.yaml, restart daemon |
| Cloud fallback when local is busy | Manual switch | Automatic via `fallbacks:` |
| Multiple clients, different models | Each client has its own URL | One URL; routing is in YAML |
| Expense tracking on cloud API calls | None | Built-in spend tracking (DB optional) |
| Model aliases ("code" = whatever is fastest) | Not possible | `model_group_alias:` in config |
| Travel / off-Mac work | Local models unreachable | Cloud fallback auto-kicks in |
| A/B test local model vs cloud | Manual | Route 20% to Groq, 80% to Ollama |

### When LiteLLM Adds Noise

| Scenario | Verdict |
|---|---|
| Single Mac, Ollama only, no cloud APIs | **Overhead not justified** — direct Ollama is simpler |
| Air-gapped machine, no internet ever | LiteLLM still works but cloud fallbacks are useless |
| Latency is critical at sub-100ms | 5–15ms proxy overhead is measurable (still negligible vs inference) |
| Minimal services philosophy | One more daemon to monitor |

### Feature Comparison

| Feature | LiteLLM (stateless) | LiteLLM (with PostgreSQL) | Ollama direct |
|---|---|---|---|
| OpenAI-compatible API | ✅ :4000 | ✅ :4000 | ✅ :11434 |
| Route to local inference | ✅ | ✅ | N/A |
| Route to cloud APIs | ✅ | ✅ | ❌ |
| Model aliasing | ✅ | ✅ | ❌ |
| Automatic fallbacks | ✅ | ✅ | ❌ |
| Load balancing across backends | ✅ | ✅ | ❌ |
| Virtual keys (per-user) | ❌ | ✅ | ❌ |
| Spend tracking | ❌ | ✅ | ❌ |
| Per-user rate limiting | ❌ | ✅ | ❌ |
| Admin web UI | ❌ | ✅ :4000/ui | ❌ |
| LaunchDaemon (this repo) | ✅ planned | ❌ not planned (PostgreSQL is too heavy) | ✅ existing |
| Setup complexity | Low | High (Postgres + Prisma migration) | Minimal |

**Decision: Stateless mode only.** PostgreSQL is not appropriate for this use case —
it's a single-node headless inference server, not a multi-tenant SaaS gateway. We will
never need per-user virtual keys or spend tracking across teams. Stateless mode gives us
everything that matters: routing, fallbacks, aliasing, and cloud access.

---

## What This Is Not

- **Not a replacement for Ollama.** Ollama must still be running for local GGUF model inference.
- **Not a replacement for Rapid-MLX.** LiteLLM routes *to* Rapid-MLX; it does not accelerate inference.
- **Not an embeddings server.** Infinity still handles `/v1/embeddings` directly. LiteLLM can proxy
  embedding requests, but it is not required to.
- **Not required for any other tool in this repo.** Everything in Phase 1 and 2 works independently.
  LiteLLM is purely additive.

## LiteLLM Configuration

### `litellm_config.yaml` — Generated by `install-tools.sh`

The proxy config is a YAML file generated from `config.json` at install time. It should be
re-generated when backends are added or removed.

**Minimal example (Ollama only):**

```yaml
model_list:
  - model_name: "ollama-default"
    litellm_params:
      model: "ollama/qwen2.5-coder:7b-instruct-q8_0"
      api_base: "http://localhost:11434"

  - model_name: "embed"
    litellm_params:
      model: "ollama/nomic-embed-text"
      api_base: "http://localhost:11434"

litellm_settings:
  drop_params: true
  request_timeout: 600
```

**Full example (Rapid-MLX primary, Ollama fallback, cloud fallback):**

```yaml
model_list:
  # Code generation — three-tier fallback
  - model_name: "code"
    litellm_params:
      model: "openai/default"          # Rapid-MLX speaks OpenAI format
      api_base: "http://localhost:8000"
      api_key: "not-needed"

  - model_name: "code-fallback-local"
    litellm_params:
      model: "ollama/qwen2.5-coder:7b-instruct-q8_0"
      api_base: "http://localhost:11434"

  - model_name: "code-fallback-cloud"
    litellm_params:
      model: "groq/llama-3.3-70b-versatile"
      api_key: "os.environ/GROQ_API_KEY"

  # Embeddings
  - model_name: "embed"
    litellm_params:
      model: "openai/default"
      api_base: "http://localhost:7997"   # Infinity
      api_key: "not-needed"

litellm_settings:
  drop_params: true
  request_timeout: 600
  fallbacks:
    - {"code": ["code-fallback-local", "code-fallback-cloud"]}

router_settings:
  routing_strategy: "least-busy"
  num_retries: 2
  timeout: 30
```

### `config.json` additions

New top-level `litellm` section to be added:

```json
{
  "litellm": {
    "enabled": false,
    "port": 4000,
    "master_key": "sk-local",
    "request_timeout": 600,
    "routing_strategy": "least-busy",
    "cloud_fallback": {
      "enabled": false,
      "groq_api_key_env": "GROQ_API_KEY",
      "anthropic_api_key_env": "ANTHROPIC_API_KEY",
      "openai_api_key_env": "OPENAI_API_KEY"
    },
    "model_aliases": {
      "code": "rapid_mlx",
      "embed": "infinity",
      "reason": "ollama"
    }
  }
}
```

- `model_aliases` maps a friendly name to which local backend handles it.
- `cloud_fallback.enabled` adds cloud providers as last-resort fallbacks.
- If `cloud_fallback.enabled: true`, the env vars must exist in the shell environment.
  `install-tools.sh` will warn but not block if they are absent.

## Implementation Plan

### Phase 3.0 — config.json Extension
**Files:** `config.json`

Add the `litellm` block. Schema:

```
litellm.enabled               bool    — whether install-tools.sh deploys LiteLLM
litellm.port                  int     — proxy listen port (default 4000)
litellm.master_key            string  — API key that clients must send (stateless auth)
litellm.request_timeout       int     — seconds before proxy gives up on a backend
litellm.routing_strategy      string  — "least-busy" | "simple-shuffle" | "latency-based"
litellm.cloud_fallback.enabled         bool
litellm.cloud_fallback.*_api_key_env   string  — env var name, NOT the key itself
litellm.model_aliases         object  — maps alias name → local backend name
```

### Phase 3.1 — `litellm_config.yaml` Generation
**Files:** `install-tools.sh` (new section)

`generate_litellm_config()` function that:

1. Reads `config.json` via `jq`
2. Writes `/Library/LiteLLM/litellm_config.yaml` from template, injecting:
   - Enabled local backends as model entries
   - `model_aliases` as `model_group_alias` entries in `router_settings`
   - `cloud_fallback` providers if enabled
   - `fallbacks` chain per alias (local primary → other local → cloud)
3. Creates `/Library/LiteLLM/` owned by `root:wheel 755`
4. Excludes `/Library/LiteLLM` from Spotlight

The YAML is generated fresh each time `install-tools.sh` runs, so it always reflects the
current state of `config.json`. Operators can hand-edit `/Library/LiteLLM/litellm_config.yaml`
after generation, but a re-run of `install-tools.sh` will overwrite it.

### Phase 3.2 — `install-tools.sh` LiteLLM Section
**Files:** `install-tools.sh`

Gated by `litellm.enabled`. Steps:

1. **Install:** `uv tool install 'litellm[proxy]'` — idempotent (uv skips if version unchanged)
2. **Resolve binary path:** `LITELLM_BIN=$(uv tool dir litellm)/bin/litellm`
3. **Generate config:** call `generate_litellm_config()`
4. **Create log dir:** `sudo mkdir -p /var/log/litellm && sudo chown root:wheel /var/log/litellm`
5. **Write plist** at `/Library/LaunchDaemons/com.litellm.proxy.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.litellm.proxy</string>
  <key>ProgramArguments</key>
  <array>
    <string>/path/to/litellm</string>
    <string>--config</string>
    <string>/Library/LiteLLM/litellm_config.yaml</string>
    <string>--port</string>
    <string>4000</string>
    <string>--detailed_debug</string>
  </array>
  <key>EnvironmentVariables</key>
  <dict>
    <key>HOME</key>
    <string>/var/root</string>
    <key>PATH</key>
    <string>/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin</string>
  </dict>
  <key>UserName</key>
  <string>root</string>
  <key>RunAtLoad</key>
  <true/>
  <key>KeepAlive</key>
  <true/>
  <key>StandardOutPath</key>
  <string>/var/log/litellm/stdout.log</string>
  <key>StandardErrorPath</key>
  <string>/var/log/litellm/stderr.log</string>
  <key>ThrottleInterval</key>
  <integer>10</integer>
</dict>
</plist>
```

6. **Bootstrap:** `launchctl bootout system ... 2>/dev/null; launchctl bootstrap system ...`
7. **Check endpoint:** `curl -s http://localhost:4000/health` after 5-second wait

**Ordering constraint:** LiteLLM must be bootstrapped after Ollama and Rapid-MLX, because
it will attempt to contact their endpoints during startup health checks.

### Phase 3.3 — `verify.sh` LiteLLM Section
**Files:** `verify.sh`

New section after Infinity, gated by `litellm.enabled`:

```
LITELLM
  ✓ daemon com.litellm.proxy running
  ✓ /health returns 200
  ✓ /v1/models lists N routes
  ✓ alias "code" resolves to rapid_mlx (or ollama)
  ! cloud_fallback configured but GROQ_API_KEY not set  [WARN]
```

- `check_http "LiteLLM health" "http://localhost:4000/health" "." 5`
- `check_http "LiteLLM models" "http://localhost:4000/v1/models" "data" 5`

### Phase 3.4 — `restore.sh` LiteLLM Section
**Files:** `restore.sh`

New block before the pmset restore:

```bash
# LiteLLM proxy
LITELLM_PLIST="/Library/LaunchDaemons/com.litellm.proxy.plist"
remove_daemon "$LITELLM_PLIST" "com.litellm.proxy"
sudo rm -rf /Library/LiteLLM
sudo rm -f /var/log/litellm/stdout.log /var/log/litellm/stderr.log
sudo rmdir /var/log/litellm 2>/dev/null || true
```

## Open Questions / Decisions Needed

### 1. Default: enabled or disabled?

**Recommendation: disabled by default** (`litellm.enabled: false`).

Rationale: LiteLLM requires at least one backend to be running before it's useful. Enabling it
by default when only Ollama is enabled would just add a passthrough hop with no benefit. Users
who want the routing/fallback features should opt in consciously.

### 2. Cloud fallback key handling

Cloud API keys must NOT go in `config.json` (which is checked into git). Options:
- **Option A:** Env var names in config.json, keys in `~/.zshrc` / `~/.bash_profile`
- **Option B:** Separate `~/.mac-llm-secrets` file that is sourced by the LaunchDaemon plist
- **Option C:** macOS Keychain lookup via `security find-generic-password`

**Recommendation: Option A** (env var names in config.json; keys set by user in shell profile).
Simpler, consistent with how most tools handle this. `install-tools.sh` warns if the named env
var is absent at install time. The LaunchDaemon plist sources the keys from the environment
at launch (they're picked up from root's environment if set in `/etc/launchd.conf` or the plist's
`EnvironmentVariables` dict). Document clearly that keys must be exported before running
`install-tools.sh`.

### 3. `litellm` binary path is dynamic

`uv tool install` places the binary in a user-specific path (`~/.local/bin/litellm` for the
user who installs it, which is root when run via `sudo install-tools.sh`). This means the plist
`ProgramArguments[0]` must be resolved dynamically at install time:

```bash
LITELLM_BIN="$(sudo -u root uv tool dir litellm 2>/dev/null)/bin/litellm"
# Fallback: /usr/local/bin/litellm (if installed globally)
[[ -x "$LITELLM_BIN" ]] || LITELLM_BIN="$(command -v litellm)"
```

The resolved path gets written into the plist. This is the same pattern used for mlx-lm and Infinity.

### 4. Port conflict risk

Port 4000 may be in use (e.g. a React dev server). `precheck.sh` should add port 4000 to its
port-availability check if LiteLLM is enabled. This is a minor `precheck.sh` addition that can
be done in Phase 3.0 alongside the config.json change.

## What This Enables (Summary)

```bash
# Day 1 — local only
# Claude Code → http://localhost:4000 → Rapid-MLX → qwen2.5-coder

# Day 2 — traveling without the Mac
# Just set GROQ_API_KEY in your environment
# Claude Code → http://localhost:4000 → (Rapid-MLX unavailable) → Groq cloud
# No client config change required

# Day 3 — switch default model
# Edit /Library/LiteLLM/litellm_config.yaml
# Change model_group_alias: { "code": "ollama-default" }
# sudo launchctl bootout system /Library/LaunchDaemons/com.litellm.proxy.plist
```
